# 实时显示与 Legacy 对齐改动说明

## 1. 文档范围

本文档描述的是：

- 当前工作区里的最新实现
- 相对于“本次聊天开始前”的版本，发生了哪些变化

也就是说，本文档不记录中间试验性改动，而只记录最终保留下来的结果。

## 2. 本次改动的总目标

相对聊天开始前的版本，本次主要完成了两类工作：

1. 给当前实验框架补上统一的实时显示能力。
2. 把当前框架中若干关键行为尽量向 legacy 脚本靠拢。

这里的 “legacy” 主要参考：

- [phase_calibration_golden_search_for_256ch_for_CCD.m](</d:/学习/26光电竞赛/OPA_algo/OPA_algo/experiment/phase_calibration_golden_search_for_256ch_for_CCD.m:1>)

### 3.1 新增的配置能力

在 [exp_default_config.m](</d:/学习/26光电竞赛/OPA_algo/OPA_algo/experiment/exp_default_config.m:44>) 中新增了 CCD 实时显示配置：

### 3.2 测量函数支持返回图像信息

在 [exp_make_measure_fn.m](</d:/学习/26光电竞赛/OPA_algo/OPA_algo/experiment/exp_make_measure_fn.m:1>) 中，测量函数从原来的单输出：

```matlab

```

升级为支持双输出：

```matlab

```

在 CCD 模式下，`sensorInfo` 当前包含：

- `sensorInfo.image`
- `sensorInfo.peakIntensity`

作用：

- 让显示模块能够拿到当前 CCD 图像
- 让显示模块能够判断是否过曝
- 为“最佳图像”记录提供输入

### 4.3 去掉整轮回退逻辑

相对聊天开始前，当前版本已经不再保留“整轮回退”。

聊天开始前：

- `mREV-GSS`
- `5PPS`
- `PFPD`
- `CAIO`

都带有按整轮进行的 rollback 逻辑。

当前版本：

- 已全部移除整轮回退
- `roundAccepted` 仍然保留在返回结构中，但当前实现中其值会保持为 `true`

原因：

- 这是你在本次聊天中明确要求去掉的行为
- 当前目标是让行为更接近 legacy 中“按通道接受”的风格，而不是“整轮失败则回退整轮”

---

### 4.4 增加 legacy 风格的单通道接受逻辑

新增文件：

- [exp_apply_legacy_channel_acceptance.m](</d:/学习/26光电竞赛/OPA_algo/OPA_algo/experiment/exp_apply_legacy_channel_acceptance.m:1>)

作用：

- 在某个通道得到候选更新结果后
- 不是直接无条件接受
- 而是再进行一次更接近 legacy 思路的接受判定

当前公共逻辑是：

1. 测量通道更新前的基线强度
2. 测量通道更新后的候选强度
3. 将候选强度与“基线强度”和“上一已接受强度”构成的阈值比较
4. 决定这一通道最终接受哪个状态

说明：

- 这是当前框架对 legacy 单通道接受行为的统一抽象
- 它比聊天开始前版本更接近 legacy
- 但由于 5PPS / PFPD / CAIO 在 legacy 脚本中没有一一对应的旧实现，因此这里只能做到“统一向 legacy 风格靠拢”，而不是宣称与某个旧脚本完全逐行等价

## 5. 当前图窗显示的最终行为

### 5.1 CCD 模式下

运行中会弹出一个图窗，默认名称为：

- `Calibration Realtime Monitor`

布局为 `1x3`：

1. 左图：强度历史曲线
2. 中图：当前 CCD 图像
3. 右图：历史最佳 CCD 图像

其中：

- 左图会显示当前算法名、轮次、通道号
- 中图会显示当前算法名、轮次、通道号
- 右图会显示：
  - `FWHM`
  - `Beam Position`
  - 最佳图像出现的轮次和通道号

另外还有这些行为：

- 当前图像超过 `saturationThreshold` 时会报警
- 图像显示使用固定色轴，减少闪烁
- 每个通道完成更新后刷新一次

---

### 5.2 非 CCD 模式下

如果传感器是：

- `DSO`
- `CAPTURE_CARD`

则显示为单图历史曲线，不显示图像子图。

## 7. 与 legacy 仍然存在的差异

虽然当前已经明显向 legacy 靠拢，但仍有几点需要如实说明。

### 7.1 返回结果仍然是结构体，不是脚本工作区

legacy 本质是脚本，结束后变量直接留在工作区。

当前版本仍然是函数化框架，因此：

- 结果通过 `result` 结构体返回
- 而不是直接把变量散落在 base workspace

不过字段命名已经尽量向 legacy 靠拢。

---

### 7.2 `V_measure` 的严格定义未必与 legacy 完全逐点一致

当前的 `V_measure` 是通过顶层包装测量函数统一记录得到的。

它能很好地反映运行过程中的测量序列，但如果要求：

- 与 legacy 脚本内部每一次具体记录时机完全逐点一致

那当前版本还不能保证 100% 完全同构。

也就是说：

- 字段名已经补齐
- 基本语义已经接近
- 但严格的记录时序定义仍然和 legacy 脚本内部写法不完全一样

---

### 7.3 `5PPS / PFPD / CAIO` 没有一个一一对应的 legacy 旧脚本

因此：

- 它们当前采用的是“向 legacy 风格靠拢”的统一行为
- 不能说与某个旧脚本完全逐行等价

真正能直接拿来做强对照的 legacy 参考主要还是：

- `phase_calibration_golden_search_for_256ch_for_CCD.m`

## 8. 当前程序运行过程中的结果

### 8.1 运行中会产生什么

程序运行期间会产生两类结果：

1. 硬件侧实际输出的电压更新
2. MATLAB 图窗中的实时显示

每个通道完成更新后：

- 电压会被写入硬件
- 随后测量当前状态
- 刷新显示
- 并记录历史强度

---

### 8.2 运行结束后会得到什么

程序运行结束后：

1. 最终 `controlU` 会再次通过 `measureFn(result.controlU)` 写入硬件并测量
2. 顶层返回 `result` 结构体
3. 结构体里包含当前算法结果、实时显示相关最佳结果、以及更接近 legacy 的若干命名字段

---

## 9. 本次聊天最终留下来的核心结论

相对聊天开始前，当前版本已经完成了这些最终改动：

1. 所有算法都支持统一实时显示。
2. 刷新时机固定为“每个通道完成更新后刷新一次”。
3. CCD 初始化补回 legacy adaptor 注册。
4. 开始前补回 legacy 风格仪器清理。
5. 整轮回退逻辑已移除。
6. 当前框架新增了更接近 legacy 的单通道接受逻辑。
7. 返回结果中补回了一组 legacy 风格命名字段。
8. `delta_V_measure` 和 `delta_V_measure_final` 最终没有保留在返回结果里。

这份文档描述的就是当前最终状态，相对本次聊天开始前的差异。
